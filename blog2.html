The two mental models

The original editor was node-first: click to drop a node, click two nodes to connect them with a capacitor / inductor / junction. It worked, but it didn't feel like drawing a schematic. People draw circuits the other way around: lay down wires, then drop components on top of them. Topology comes from the wires; nodes are wherever wires meet.

So we kept the old mode and added a new one alongside it — a wire mode, toggled at the top of the canvas. The change isn't cosmetic: it inverts the data model, the click dispatch, and even what "a node" means. Electrical nodes are no longer placed by the user; they're derived from the wire graph.

Why bother with two modes?
The schematic mode is still the fastest way to express tiny abstract topologies (a single transmon, a two-node coupler). Wire mode wins as soon as you have any branching wire that more than one component sits on. Keeping both means we never have to argue about which one is "right."

A new data model: vertices, segments, components

The wire graph is three flat arrays. Vertices are wire turning-points (not electrical nodes). Segments are pure-wire connections between two vertices — always type 'wire'. Components are an overlay: a component sits on a segment at parameter t ∈ [0,1] and electrically breaks the wire there, but leaves the segment intact in the data model.

src/wire/wireModel.js 
export const EMPTY_WIRE = {
  vertices:   [],   // { id, x, y }
  segments:   [],   // { id, from, to }              — always pure wire
  components: [],   // { id, segmentId, t, type, value }  — overlay
  nextVertexId: 0, nextSegmentId: 0, nextComponentId: 0,
};

The single most important consequence of that split: deleting a component leaves the wire intact. You can place a junction, change your mind, delete it, and the wire under it is still there. In the old model a component was the edge; deleting it ripped a hole in the topology.

Hover-driven click dispatch

In schematic mode, every node and edge had its own onClick handler. That doesn't translate — in wire mode you can click on empty space, on a grid intersection, on the middle of a segment, on an existing vertex, or on a component, and the meaning of the click depends on which tool is active. Adding handlers to every visual would push a lot of duplicate logic into each one.

Instead there is a single canvas-level onClick, and a single computeHover() function that resolves the cursor position to a hover descriptor — "what would happen if you click here right now." The click handler reads the descriptor and acts on it.

src/App.jsx (excerpt) 
// Hover priority: existing vertex → existing segment → grid / free.
const computeHover = useCallback((pt, shiftKey) => {
  if (mode !== 'wire') return null;

  const vid = snapToVertex(verts, pt.x, pt.y);
  if (vid !== null) return { kind: 'vertex', id: vid, ... };

  if (!selectedTool) {
    const compHit = snapToComponent(wire, pt.x, pt.y);
    if (compHit) return { kind: 'component', id: compHit.id };
  }

  const segHit = snapToSegment(verts, segs, pt.x, pt.y);

  if (selectedTool === 'wire') {
    if (segHit) return { kind: 'segment', segmentId: segHit.id, t: segHit.t, ... };
    if (shiftKey) return { kind: 'free', x: pt.x, y: pt.y };
    const g = snapToGrid(pt.x, pt.y);
    return { kind: 'grid', x: g.x, y: g.y };
  }

  if (selectedTool === 'C' || selectedTool === 'L' || selectedTool === 'JJ') {
    if (!segHit) return null;
    const f = snapFraction(segHit.t);
    return { kind: 'componentOnSegment', segmentId: segHit.id, t: f.t, ... };
  }
  ...
}, [mode, selectedTool, wire]);

The descriptor doubles as a render input — WireLayer reads the same hover and paints the ghost preview (a dashed circle at a grid intersection, a translucent component at a fraction snap, etc.). Click logic and preview can never disagree because they're driven by the same value.

Snap-to-grid with Shift override

Wire mode uses a 30 px grid. The cursor magnetically snaps to the nearest grid intersection unless Shift is held, in which case it places freely. The default-snap, opt-out-with-modifier rule is the one schematic editors converge on for a reason: 95% of the time you want a clean lattice; the other 5% you want pixel control.

src/wire/wireModel.js 
export const GRID = 30;

export function snapToGrid(x, y, grid = GRID) {
  return { x: Math.round(x / grid) * grid, y: Math.round(y / grid) * grid };
}

The same rule applies while dragging an existing vertex: hold Shift to break the snap. Tiny detail, but it means the user never has to learn two different grid behaviors.

Branching from anywhere on a wire

When the wire tool is active and you hover over an existing segment, the descriptor's kind is 'segment'. Clicking there has to insert a new vertex on that segment — the segment splits into two halves, and any components on it get reassigned to whichever half they fall into, with their t rescaled.

src/wire/wireModel.js 
export function splitSegmentAt(wire, segmentId, t) {
  const seg = wire.segments.find((s) => s.id === segmentId);
  ...
  const halfA = { id: halfAId, from: seg.from, to: vid };
  const halfB = { id: halfBId, from: vid, to: seg.to };

  const newComponents = wire.components.map((c) => {
    if (c.segmentId !== segmentId) return c;
    if (c.t <= t) return { ...c, segmentId: halfAId, t: t > 0 ? c.t / t : 0 };
    return         { ...c, segmentId: halfBId, t: t < 1 ? (c.t - t) / (1 - t) : 1 };
  });
  ...
}

A "ghost vertex" preview marks the would-be split point as you hover, so branching feels like a deliberate action rather than an arbitrary canvas click. Because segments and components are independent, the split is a pure topology change — no electrical-node reshuffling happens here. That comes later, automatically.

Components as overlays

The renderer is where the overlay model earns its keep. A segment with two components on it is drawn as three separate wire pieces — before the first component, between the two components, after the second — with the components painted on top of the gaps. Each piece is colored by the electrical node it belongs to, so the user sees the topology at a glance.

src/components/WireLayer.jsx (excerpt) 
// Build slot intervals + node ids:
// slot 0: from-vertex node — wire stretch before first component.
// slot k (1..N-1): phantom between comps[k-1] and comps[k].
// slot N: to-vertex node.
const slots = [];
if (compsOnSeg.length === 0) {
  slots.push({ t1: 0, t2: 1, nodeId: vertexNodeId.get(s.from) });
} else {
  slots.push({ t1: 0, t2: compsOnSeg[0].t - halfT,
               nodeId: vertexNodeId.get(s.from) });
  for (let k = 1; k < compsOnSeg.length; k++) {
    const phantomKey = `gap:${compsOnSeg[k - 1].id}:${compsOnSeg[k].id}`;
    slots.push({ t1: compsOnSeg[k - 1].t + halfT,
                 t2: compsOnSeg[k].t - halfT,
                 nodeId: phantomNodeIdByKey.get(phantomKey) });
  }
  slots.push({ t1: compsOnSeg[N-1].t + halfT, t2: 1,
               nodeId: vertexNodeId.get(s.to) });
}
Why the half-length offset?

Each component has a fixed visual length (60 px). The wire pieces stop a half-length short of each component center on either side, so the symbol's body sits cleanly in the gap with no overlapping line underneath. halfT = halfLen / segmentLength converts that into t-space so the math works on diagonal segments too.

The same CircuitSymbol component the schematic mode uses gets reused verbatim for the overlay. The only new thing wire mode adds is the slot math.

Magnetic fraction snaps

When dropping a component on a segment, the cursor's t magnetically snaps to one of 1/4, 1/3, 1/2, 2/3, 3/4. A small floating badge shows the fraction it snapped to. It is a tiny detail, but it is the difference between diagrams that look hand-laid and diagrams that look like someone forgot the alignment guides.

src/wire/wireModel.js 
export const FRACTIONS = [
  { t: 1 / 4, label: '1/4' },
  { t: 1 / 3, label: '1/3' },
  { t: 1 / 2, label: '1/2' },
  { t: 2 / 3, label: '2/3' },
  { t: 3 / 4, label: '3/4' },
];

export function snapFraction(t, tol = 0.06) {
  let best = null;
  for (const f of FRACTIONS) {
    const d = Math.abs(t - f.t);
    if (d < tol && (best === null || d < best.d)) best = { ...f, d };
  }
  return best ? { t: best.t, label: best.label } : { t, label: null };
}

Auto-detecting electrical nodes in real time

This is the core of the redesign. The user never places electrical nodes — they place wires and components, and the editor figures out the nodes. Two vertices share an electrical node iff they are connected by a path of segments that carry no components. Union-find collapses the wire graph into equivalence classes; each class becomes a node.

src/wire/wireModel.js (excerpt) 
// Union-find over vertices, only via segments with no components.
const parent = new Map();
vertices.forEach((v) => parent.set(v.id, v.id));
const find = (id) => { /* path-compressed find */ };
const union = (a, b) => { /* union by setting parent */ };

for (const s of segments) {
  if (!compsBySegment.has(s.id)) union(s.from, s.to);
}

Components on the same segment are treated as in series. A segment with N components creates N+1 electrical nodes along its length: the from-vertex's node, the to-vertex's node, and N−1 phantom nodes between adjacent components. Phantom nodes have no vertex of their own — they only exist in the analysis output.

The phantom-node walk

    rightNodeId = vB;                      // to-vertex's node
  } else {

    // ...allocate a new phantom node, position it at midpoint...

Detection runs on every wire mutation via a single useEffect, with a ref carrying the previous result so the effect doesn't depend on its own output:

  setWireNodes(result.nodes);
  setWireEdges(result.edges);
}, [wire]);

Stable identity across edits

Auto-detection regenerates nodes from scratch on every mutation. That's fine for the topology, but it would be miserable if the user's customizations (a custom label, a custom color, "this one is ground") got reshuffled every time they added a wire. Each node needs a stable identity that survives unrelated edits.

Two carry-forward keys do this. For real (vertex-backed) nodes the key is the smallest vertex id in the equivalence class — an "anchor." Add a vertex to the class and the anchor doesn't change. Remove the anchor and a new one takes over, but unrelated nodes are untouched. For phantom nodes the key is the pair of flanking component ids, since those define the gap.

src/wire/wireModel.js (excerpt) 
const carry = prevByAnchor.get(anchor);
let label = null;
let userLabel = false;
if (carry?.userLabel && carry.label) {
  label = carry.label;
  userLabel = true;
  usedLabels.add(label);
}
const color = carry?.userColor
  ? carry.color
  : NODE_PALETTE[paletteIndexForKey(`v:anchor:${anchor}`)];

Auto-assigned labels (\phi_0, \phi_1, ...) reflow as topology changes; user-set labels stick. The userLabel / userColor flags are the dividing line. Default colors are picked by hashing the anchor key into a palette, so unrelated parts of the graph don't shuffle when one part changes.

The unique-\phi allocator 
// Final pass: assign auto labels, skipping any \phi_{k} already in use.
let autoIdx = 0;
for (const n of nodes) {
  if (n.label) continue;
  while (usedLabels.has(`\\phi_{${autoIdx}}`)) autoIdx++;
  n.label = `\\phi_{${autoIdx}}`;
  usedLabels.add(n.label);
  autoIdx++;
}

The user might rename one of their nodes to \phi_5; the allocator skips that index when handing out auto labels, so we never end up with two nodes both showing \phi_5.

Right-click, drag, polish

Three small things that disproportionately affect how the editor feels:

  • Right-click clears the active tool. No "select the cursor tool to deselect" minigame — right-click anywhere on the canvas drops you back to selection mode.
  • Wire vertices grab on mousedown and snap to the grid as you drag (or move freely with Shift). The same grid logic the placement code uses.
  • Hover preview always renders the descriptor. A ghost vertex at the grid snap, a dashed crosshair on the intersection, a translucent component at the fraction snap. You see the click before you commit it.
Single source of hover truth
The fact that previews and click logic both read from computeHover() means every "what would happen if I click here" is shown before you click. There is no UI state that lives only inside the click handler.

Symbolic values

Components in wire mode hold LaTeX strings, not numbers. A new capacitor is born as C_{0}, the next as C_{1}, the next junction as E_J^{0}. The matrices the analysis panel renders are then real symbolic expressions — C_0 + C_1 on a diagonal, -C_0 off-diagonal, and so on.

src/wire/wireModel.js 
export function nextComponentSymbol(components, type) {
  const sym = ELEMENT_TYPES[type].symbol;       // 'C' | 'L' | 'E_J'
  const used = new Set(components.map((c) => c.value));
  let k = 0, candidate;
  do {
    candidate = type === 'JJ' ? `E_J^{${k}}` : `${sym}_{${k}}`;
    k++;
  } while (used.has(candidate));
  return candidate;
}

Renaming through the properties panel just sets component.value to whatever LaTeX the user types. The allocator skips anything currently in use, so renaming C_2 to C_a frees up C_2 for the next auto-named capacitor.

The capacitance matrix, symbolically

With symbolic values flowing all the way through the edge list, the capacitance matrix is built directly from them. Each cell is an array of signed terms; a tiny formatter renders the array as LaTeX.

src/physics/hamiltonian.js 
export function capacitanceMatrix(nodes, edges) {
  const sorted = [...nodes].sort((a, b) => a.id - b.id);
  const idx = {};
  sorted.forEach((n, i) => (idx[n.id] = i));
  const cells = Array.from({ length: N }, () =>
    Array.from({ length: N }, () => []));

  for (const e of edges) {
    if (e.type !== 'C') continue;
    const i = idx[e.from], j = idx[e.to];
    if (i === j) continue;
    const term = String(e.value);
    cells[i][i].push({ sign: '+', term });
    cells[j][j].push({ sign: '+', term });
    cells[i][j].push({ sign: '-', term });
    cells[j][i].push({ sign: '-', term });
  }
  return { cells, nodeList: sorted };
}

export function formatSymbolicSum(parts) {
  if (parts.length === 0) return '0';
  return parts.map((p, i) => {
    if (i === 0) return p.sign === '-' ? `-${p.term}` : p.term;
    return p.sign === '-' ? ` - ${p.term}` : ` + ${p.term}`;
  }).join('');
}

The analysis panel then shows the adjacency matrix and the capacitance matrix side by side: counts on the left, symbols on the right, both regenerated live as the wire graph changes.

A two-panel layout

The single right-side panel was getting cramped. Wire mode introduces an editable list of identities — the auto-detected node names, colors, and ground flags — that didn't sit naturally next to capacitance matrices and Hamiltonians. So we split:

  • Left panel — NodeLabelsPanel: one row per electrical node, with rename / color / ground controls. This is the answer to "who am I looking at?"
  • Right panel — HamiltonianPanel: adjacency, capacitance, energies, exported parameters. The answer to "what does it compute?"

The canvas sits in the middle. Identity on one side, physics on the other, drawing surface in between. Reading left-to-right matches how you actually think about the circuit: name the nodes, draw the topology, read off the matrices.

Undo/redo that survives a drag

A wire-first editor makes more small mutations than the schematic version did — every grid click is a vertex, every component drop is an overlay insert, every segment hover-and-click is a split. Without undo, one stray click on a complex diagram is a disaster. With naive undo, dragging a vertex 80 pixels writes 80 history entries and "undo" feels broken.

The hook keeps both modes' state under one roof and pauses recording for the duration of a drag, so a drag collapses into a single entry. A snapshot ref on each render diffs against the current snapshot before pushing, so no-op rerenders don't pollute history. The stack is capped at 100.

src/hooks/useHistory.js (excerpt) 
useEffect(() => {
  if (skipRef.current) { skipRef.current = false; sync(); return; }
  if (dragging !== null) return;                    // pause during drags

  const h = historyRef.current;
  const cur = h.stack[h.index];
  if (cur && sameSnapshot(cur, nodes, edges, wire, wireNodes, wireEdges)) return;

  h.stack = h.stack.slice(0, h.index + 1);
  h.stack.push(snapshot(nodes, edges, wire, wireNodes, wireEdges));
  h.index = h.stack.length - 1;
  if (h.stack.length > MAX_HISTORY) { h.stack.shift(); h.index--; }
  sync();
}, [nodes, edges, wire, wireNodes, wireEdges, dragging, sync]);

The skipRef flag exists because restoring a snapshot triggers the same effect that records snapshots. Without the skip, undo would push the just-restored state as a new history entry and redo would never work.

Pan, zoom, and resizable panels

A single-screen editor was fine for a transmon. As soon as you draw a multi-loop fluxonium with a half-dozen named nodes the canvas runs out of room. Three small viewport tools fix it without reaching for a layout library:

  • Pan by middle-click or background drag, with a 3 px deadzone so a normal click doesn't accidentally start a pan.
  • Zoom with the wheel, clamped to [0.1, 10], anchored on the cursor so the point under the mouse stays fixed.
  • Fit view — one button auto-zooms and centers the bounding box of all nodes with padding.

Both side panels are independently resizable. The left (NodeLabelsPanel) and right (HamiltonianPanel) each have their own drag controller (useResizablePanel) clamped to [240, 900] px. Symbolic capacitance matrices grow wide with circuit size; the right panel needs to grow with them.

Keyboard ergonomics

Tool-driven editors live or die by their keyboard layer. The shortcuts here are deliberately boring — the same ones every drawing app uses, so there is nothing new to learn:

  • Ctrl+Z / Ctrl+Shift+Z / Ctrl+Y — undo / redo.
  • Esc — clear the active tool, then deselect (two presses to fully reset).
  • Delete / Backspace — remove the current selection.
  • Shift — the universal "break the snap" modifier (placement, drags, fraction snaps).

The handler refuses to fire while focus is inside an <input> or <textarea>, so renaming a node to \phi_{\text{a}} doesn't accidentally undo the last topology change.

A small companion in the toolbar: a show/hide labels toggle that suppresses the symbolic component names (C_0, E_J^1, ...) on the canvas. Useful when you're laying down a dense diagram and want the geometry to read first.

JSON round-trip with full geometry

Export is the moment of truth for a data model: anything you can't serialize back into the same diagram is a bug in the model. The wire-first model serializes cleanly because it already is three flat arrays. The export bundles the analysis-level nodes / edges (what downstream solvers consume), the full wire_geometry (vertices, segments, components with their t), and a small node_overrides list capturing only the user-customized labels, colors, and ground flags.

src/circuit/exportJSON.js (excerpt) 
if (extra.wire) {
  payload.wire_geometry = {
    vertices: extra.wire.vertices.map((v) => ({ id: v.id, x: v.x, y: v.y })),
    segments: extra.wire.segments.map((s) => ({ id: s.id, from: s.from, to: s.to })),
    components: extra.wire.components.map((c) => ({
      id: c.id, segment_id: c.segmentId, t: c.t, type: c.type, value: c.value,
    })),
    next_vertex_id: extra.wire.nextVertexId, ...
  };

  if (extra.wireNodes) {
    payload.node_overrides = extra.wireNodes
      .filter((n) => n.userLabel || n.userColor || n.isGround)
      .map((n) => ({
        anchor: n.vertexIds?.length ? Math.min(...n.vertexIds) : null,
        phantom_key: n.phantomKey || null,
        label: n.userLabel ? n.label : undefined,
        color:  n.userColor ? n.color : undefined,
        is_ground: n.isGround || undefined,
      }));
  }
}

Import auto-detects the format: a payload with wire_geometry rebuilds the wire model and reapplies overrides keyed by anchor (smallest vertex id) and phantom key (the same identity scheme auto-detection uses). A payload without it is treated as a schematic export and falls back to circle layout if no node_positions are provided. One file format covers both modes.

Why store only overrides, not all node state?
Auto-detection regenerates nodes deterministically from the wire graph, so storing every node's auto-assigned \phi_k would be redundant — and worse, it would freeze label assignments in the file even if the graph topology changes after import. Persisting only what the user explicitly set lets the rest reflow.

What's next

Multi-component series with virtual mid-segment nodes already work. The remaining items live in the gap between "topology" and "Hamiltonian numbers":

  • External flux Φ_ext per loop. Detect cycles in the wire graph that contain at least one junction; let the user attach a flux to each one.
  • Gate charge n_g per node. Per-node offset that enters as a shift in the conjugate number operator.
  • Numeric-substitution toggle. Today the matrices are symbolic. A toggle would let the user enter actual GHz / fF / nH values and read back numerical capacitance, charging energies, and a fully evaluated Hamiltonian.
The unifying lens
All three sit on top of the same wire-first foundation. Flux loops are cycles in the segment graph; gate charges are per-node attributes already carried through the auto-detect; numeric substitution is just a second formatter on the existing symbolic terms. None of them ask the data model to change — that is the point of having gotten the model right.
