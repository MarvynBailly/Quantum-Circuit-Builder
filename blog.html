Scaffolding with Vite

We started with a clean Vite + React scaffold. Vite gives us instant hot module replacement (HMR) during development and optimized production builds, with almost zero configuration.

Terminal 
npm create vite@latest fluxonium-builder -- --template react
cd fluxonium-builder
npm install
npm install katex

The only non-React dependency is KaTeX for rendering LaTeX math. No diagram library, no state management library, no CSS framework. The whole app is just React + SVG.

vite.config.js

Minimal config — just the React plugin and a custom dev server port.

import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  server: {
    port: 3000,
    open: true,
  },
  build: {
    sourcemap: true,
  },
})

  "preview": "vite preview",  // serve the production build locally
  "lint":    "eslint src --ext .js,.jsx"
}

Project structure

We organized the source into three domains: components (React UI), circuit (element types, presets, export), and physics (scaffolded for future Hamiltonian extraction). This separation means the physics engine can be developed and tested independently of the UI.

src/

  App.jsx ← Root component (~570 lines, all the action)
  index.css ← CSS variables, dark/light theme
  components/

    SvgLatex.jsx ← KaTeX inside SVG

    hamiltonian.js ← Adjacency matrix builder (+ planned energy extraction)
    loops.js ← Flux loop detection (for future φ_ext support)
Design choice: no diagram library

We deliberately avoided libraries like React Flow or D3. Our circuits are small (under 50 nodes), and a bespoke SVG approach gives us full control over rendering, hit-testing, and future physics integration — all without fighting a library's abstraction.

The HTML entry point

The entire app loads from a single, minimal HTML file. It preloads a monospace font, sets the dark background so there's no flash of white, and mounts the React root.

index.html

  <title>Quantum Circuit Builder</title>
  <link rel="preconnect" href="https://fonts.googleapis.com" />
  <link href="https://fonts.googleapis.com/css2?family=JetBrains+Mono:wght@400;600&display=swap"

  <div id="root"></div>
  <script type="module" src="/src/main.jsx"></script>

The type="module" attribute is key — it tells the browser to treat the script as an ES module, which Vite intercepts in dev mode to serve transformed source with HMR.

The toggle itself is one line of JavaScript in App.jsx:

Wiring up React

The React entry point is trivially small. It imports KaTeX's CSS, our app styles, and mounts the root component.

src/main.jsx 
import React from 'react'
import ReactDOM from 'react-dom/client'
import 'katex/dist/katex.min.css'
import './index.css'
import App from './App'

ReactDOM.createRoot(document.getElementById('root')).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
)

All application state lives inside App.jsx via useState and useRef. We don't use Redux, Zustand, or any external state manager — at this scale, lifting state into the root component is simpler and sufficient.

The SVG canvas

The circuit editor is a single <svg> element that fills the left side of the screen. Everything — nodes, wires, symbols, labels — is rendered as SVG elements inside a transform group. 

<svg ref={svgRef} width="100%" height="100%"
     onMouseDown={handleCanvasMouseDown}
     onMouseMove={handleMouseMove}
     onMouseUp={handleMouseUp}
     onWheel={handleWheel}>

  <defs>
    <pattern id="grid" width="30" height="30" patternUnits="userSpaceOnUse">
      <circle cx="15" cy="15" r="0.8" fill="var(--grid-dot)" />
    </pattern>
  </defs>

  <!-- Background grid (fixed, not affected by pan/zoom) -->
  <rect width="100%" height="100%" fill="var(--bg-secondary)" />
  <rect width="100%" height="100%" fill="url(#grid)" />

  <!-- All circuit elements live in this transformed group -->
  <g transform={`translate(${pan.x},${pan.y}) scale(${zoom})`}>
    {/* edges drawn first (below nodes) */}
    {edges.map(edge => ...)}

    {/* nodes drawn on top */}
    {nodes.map(node => ...)}
  </g>
</svg>
Why SVG over Canvas API?

Every SVG element is a DOM node, which gives us free hit-testing (click events on individual circles and paths), easy styling via CSS variables, and simple integration with React's declarative model. For a circuit with tens of nodes, performance is a non-issue.

Nodes — placing and dragging

Each node is an object with an id, x/y position, label, and a boolean isGround flag. When the "Node" tool is active, clicking the canvas creates one: 

// On canvas click with node tool selected:
const pt = screenToSvg(e.clientX, e.clientY)
setNodes(prev => [...prev, {
  id: nextNodeId.current++,
  x: pt.x,
  y: pt.y,
  label: `\\phi_{${id}}`,   // LaTeX label
  isGround: false,
}])

The screenToSvg helper converts pixel coordinates to SVG-space by accounting for the current pan and zoom: 

function screenToSvg(clientX, clientY) {
  const rect = svgRef.current.getBoundingClientRect()
  return {
    x: (clientX - rect.left - pan.x) / zoom,
    y: (clientY - rect.top  - pan.y) / zoom,
  }
}

Dragging is handled with three mouse events. On mousedown on a node, we set dragging = node.id. On mousemove, we update that node's position. On mouseup, we clear the drag state. Try it below:

How nodes render in SVG

Each node is a group containing a circle, its label, and (for ground nodes) a three-line ground symbol:

<g key={node.id}>
  {/* Selection ring (shown when selected) */}
  {selected?.id === node.id &&
    <circle cx={x} cy={y} r={22}
      stroke="var(--accent-blue)" strokeDasharray="4 3" />}

  {/* Node circle */}
  <circle cx={x} cy={y} r={16}
    fill={node.isGround ? 'var(--ground-fill)' : 'var(--node-fill)'}
    stroke={node.isGround ? 'var(--ground-stroke)' : 'var(--accent-blue)'}
    strokeWidth={2} />

  {/* Label (rendered with SvgLatex for LaTeX support) */}
  <SvgLatex x={x} y={y - 26} text={node.label} />

  {/* Ground symbol */}
  {node.isGround && <GroundSymbol x={x} y={y + 16} />}
</g>

Edges — circuit element wiring

Connecting two nodes with a circuit element (capacitor, inductor, or Josephson junction) is a two-click workflow:

  1. Select the element tool (C, L, or JJ) from the toolbar.
  2. Click the source node — it highlights with a pulsing ring and the toolbar shows "Click target node..."
  3. Click the target node — an edge is created between them.
// On second click (target node):
setEdges(prev => [...prev, {
  id:    nextEdgeId.current++,
  from:  connectFrom,
  to:    node.id,
  type:  selectedTool,               // 'C', 'L', or 'JJ'
  value: ELEMENT_TYPES[selectedTool].defaultValue,
}])
setConnectFrom(null)  // reset for next connection

Each edge stores a type and a value (capacitance in fF, inductance in nH, or Josephson energy in GHz). The edge is rendered as a line between the two node positions, with the circuit symbol drawn at the midpoint.

Edge click hit area trick

Thin lines are hard to click. We render an invisible fat line (14px stroke, fully transparent) on top of each edge to catch clicks, while the visible line stays at 2px: 

<!-- Invisible wide hit area -->
<line x1={x1} y1={y1} x2={x2} y2={y2}
  stroke="transparent" strokeWidth={14}
  style={{ cursor: 'pointer' }}
  onClick={() => setSelected(edge)} />

<!-- Visible wire -->
<line x1={x1} y1={y1} x2={x2} y2={y2}
  stroke="var(--wire-color)" strokeWidth={2} />

Pan, zoom, and view fitting

The canvas supports mouse-wheel zoom and click-drag panning. Both manipulate the pan and zoom state, which feed into the SVG group's transform.

Zoom (mouse wheel)

We use a gentle 1.04× factor per scroll tick (tuned down from the default 1.1× which felt too jumpy). The zoom is anchored to the cursor position so the point under the mouse stays fixed:

function handleWheel(e) {
  e.preventDefault()
  const factor = e.deltaY < 0 ? 1.04 : 1 / 1.04
  const newZoom = clamp(zoom * factor, 0.1, 10)

  // Keep the point under the cursor fixed
  const rect = svgRef.current.getBoundingClientRect()
  const mx = e.clientX - rect.left
  const my = e.clientY - rect.top

  setPan({
    x: mx - (mx - pan.x) * (newZoom / zoom),
    y: my - (my - pan.y) * (newZoom / zoom),
  })
  setZoom(newZoom)
}
Pan (middle-click or background drag)

Clicking and dragging the background (not a node) pans the canvas. We track the start position in a ref to avoid re-renders on every pixel of movement:

// On mouse down on canvas background:
panInfo.current = { startX: e.clientX, startY: e.clientY, panX: pan.x, panY: pan.y }

// On mouse move while panning:
setPan({
  x: panInfo.current.panX + (e.clientX - panInfo.current.startX),
  y: panInfo.current.panY + (e.clientY - panInfo.current.startY),
})
Fit view

The "Fit" toolbar button computes a bounding box around all nodes and scales/translates to fit them in the viewport with 80px padding:

function resetView() {
  if (nodes.length === 0) return
  const xs = nodes.map(n => n.x), ys = nodes.map(n => n.y)
  const [minX, maxX] = [Math.min(...xs), Math.max(...xs)]
  const [minY, maxY] = [Math.min(...ys), Math.max(...ys)]

  const pad = 80
  const w = svgRef.current.clientWidth
  const h = svgRef.current.clientHeight
  const scale = Math.min(w / (maxX - minX + pad*2),
                         h / (maxY - minY + pad*2), 2)

  setZoom(scale)
  setPan({
    x: (w - (maxX + minX) * scale) / 2,
    y: (h - (maxY + minY) * scale) / 2,
  })
}

LaTeX labels with KaTeX

Node labels like \phi_{1} should render as proper mathematical notation. We use KaTeX for this, but embedding LaTeX inside SVG requires a trick.

SVG doesn't natively support HTML content. The workaround is <foreignObject>, which lets us embed an HTML <div> inside the SVG tree:

src/components/SvgLatex.jsx (simplified) 
import katex from 'katex'

export default function SvgLatex({ x, y, text }) {
  const isLatex = /[\\_{^}]/.test(text)

  if (!isLatex) return <text x={x} y={y}>{text}</text>

  const html = katex.renderToString(text, { throwOnError: false })

  return (
    <foreignObject x={x - 60} y={y - 16} width={120} height={32}>
      <div style={{ textAlign: 'center' }}
           dangerouslySetInnerHTML={{ __html: html }} />
    </foreignObject>
  )
}

The same approach is used in InlineLatex.jsx for the right-side panel, but there we can render KaTeX directly into a <span> since we're in HTML context.

Detecting LaTeX

We use a simple regex (/[\\_{^}]/) to decide whether a label contains LaTeX syntax. Plain text labels like "GND" fall through to a regular SVG <text> element.

The properties panel

Clicking a node or edge opens a floating panel in the top-right of the canvas. This panel lets you edit the selected element's properties.

For nodes: edit the label (supports LaTeX), toggle the ground flag, or delete the node.

For edges: see the element type (color-coded), edit the value, see the unit (fF / nH / GHz), or delete the edge.

src/components/PropertiesPanel.jsx (structure) 
export default function PropertiesPanel({ selected, nodes, edges,
                                           onUpdateNode, onUpdateEdge,
                                           onDeleteNode, onDeleteEdge }) {
  if (!selected) return null

  const isNode = selected.type === 'node'

  return (
    <div style={{ position: 'absolute', top: 12, right: 12, ... }}>
      {isNode ? (
        <>
          <input value={node.label} onChange={...} />
          <label><input type="checkbox" /> Ground node</label>
          <button onClick={onDeleteNode}>Delete</button>
        </>
      ) : (
        <>
          <span style={{ color: ELEMENT_TYPES[edge.type].color }}>
            {ELEMENT_TYPES[edge.type].label}
          </span>
          <input type="number" value={edge.value} onChange={...} />
          <span>{ELEMENT_TYPES[edge.type].unit}</span>
          <button onClick={onDeleteEdge}>Delete</button>
        </>
      )}
    </div>
  )
}

Pressing Escape deselects whatever is selected and also cancels a pending connection.

The analysis panel & adjacency matrix

The right sidebar (HamiltonianPanel.jsx) shows the circuit's graph connection matrix — a table where each cell counts how many circuit elements connect two nodes. This is the first step toward full Hamiltonian extraction: the topology has to be right before we can stamp physical values into a capacitance matrix.

src/physics/hamiltonian.js 
export function adjacencyMatrix(nodes, edges) {
  const n = nodes.length
  const matrix = Array.from({ length: n },
                   () => Array(n).fill(0))

  for (const edge of edges) {
    const i = nodes.findIndex(n => n.id === edge.from)
    const j = nodes.findIndex(n => n.id === edge.to)
    if (i >= 0 && j >= 0) {
      matrix[i][j]++
      matrix[j][i]++   // symmetric
    }
  }

  return { matrix, nodeIds: nodes.map(n => n.id) }
}

Undo/redo

We implemented a simple but effective snapshot-based undo/redo system without any library.

[state 0] → [state 1] → [state 2] → [state 3]
                      ↑ undo       ↑ current

The history lives in a useRef (not state, to avoid re-render loops):

const historyRef = useRef({
  stack: [{ nodes: initialNodes, edges: initialEdges }],
  index: 0,
})
const skipHistoryRef = useRef(false)

A useEffect watches nodes and edges. Whenever they change (and we're not currently restoring a snapshot), it pushes a new snapshot onto the stack: 

useEffect(() => {
  if (skipHistoryRef.current) {
    skipHistoryRef.current = false
    return
  }
  // Compare to last snapshot via JSON (skip if identical)
  const last = history.stack[history.index]
  if (JSON.stringify({ nodes, edges }) === JSON.stringify(last)) return

  // Truncate any redo entries, push new snapshot
  history.stack = history.stack.slice(0, history.index + 1)
  history.stack.push({ nodes: structuredClone(nodes),
                       edges: structuredClone(edges) })
  history.index++

  // Cap at 100 snapshots
  if (history.stack.length > 100) {
    history.stack.shift()
    history.index--
  }
}, [nodes, edges])

Ctrl+Z decrements the index and restores that snapshot. Ctrl+Y (or Ctrl+Shift+Z) increments it. The skipHistoryRef flag prevents the restoration itself from creating a new history entry. Here's a miniature version of the same mechanism:

Export, presets, and what's next

The toolbar's Export JSON button serializes the current circuit into a downloadable file containing nodes, edges, and the adjacency matrix:

src/circuit/exportJSON.js 
export function downloadJSON(nodes, edges, filename = 'circuit-topology.json') {
  const payload = {
    _meta: { generator: 'fluxonium-builder', exported_at: new Date() },
    nodes: nodes.map(n => ({ id: n.id, label: n.label, is_ground: n.isGround })),
    edges: edges.map(e => ({
      id: e.id, from: e.from, to: e.to,
      type: e.type, value: e.value,
      unit: ELEMENT_TYPES[e.type].unit,
    })),
    adjacency_matrix: adjacencyMatrix(nodes, edges).matrix,
  }

  const blob = new Blob([JSON.stringify(payload, null, 2)],
                        { type: 'application/json' })
  const url = URL.createObjectURL(blob)
  const a = document.createElement('a')
  a.href = url
  a.download = filename
  a.click()
  URL.revokeObjectURL(url)
}

The circuit presets module (presets.js) defines template topologies for common superconducting qubits (transmon, fluxonium, JJ arrays). Each preset is a factory function returning a set of nodes and edges that can be loaded onto the canvas.

What's next: the physics engine

Right now the app gives us the circuit topology — nodes, edges, and the graph connection matrix. The next phase is building out the src/physics/ module to extract the full Hamiltonian parameters from this topology:

  • Capacitance matrix via modified nodal analysis (MNA stamping)
  • Charging energies EC = e²/(2C) per node
  • Inductive energies EL = φ0²/L per inductor
  • Josephson energies EJ (already stored in GHz)
  • Flux loop detection for external flux φext parameters
  • Symbolic Hamiltonian display in the phase/number basis
  • Export to scQubits / QuTiP format for downstream simulation

The foundations are already laid — constants.js has the SI physical constants, linalg.js has matrix inversion, and loops.js has the flux loop detection algorithm. Wiring these into the UI is the next step.
